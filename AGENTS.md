# Repository Guidelines

## Project Structure
KGsAuto is a Python/React knowledge-graph system. FastAPI apps live in `apps/`: `apps/graph_api` is the main Graph/Admin API and `apps/chat_api` is the product chatbot API over `services/rag_system`. Core packages are in `services/`: `entity_resolution`, `rag_system`, `llms`, `crawler`, `extraction`.

- `services/rag_system/`: Unified RAG with `pipeline.py` (main entry), `modes/` (semantic_search, graph_search, naive_grag, hybrid), `storage/`, `retrieval/`, `graph/`, `workflows/graph_search/`, `evaluation/`. Read `services/rag_system/README.md` for detailed architecture.
- `services/extraction/`: Extracts knowledge graphs from documents via CLI (`python -m services.extraction.cli`).
- `services/entity_resolution/`: 3-stage entity resolution pipeline via CLI (`python -m services.entity_resolution.cli --stage all`).
- `apps/frontend/src/`: Vite React UI with components in `components/` and pages in `pages/`.
- `data/`: Generated artifacts and extracted data.

```bash

# Run with qdrant backend for persistence between stages
python -m services.entity_resolution.cli --stage stage1 --input-dir data/extracted --store-backend qdrant --run-id my_run
python -m services.entity_resolution.cli --stage stage2 --store-backend qdrant --run-id my_run
python -m services.entity_resolution.cli --stage stage3 --store-backend qdrant --run-id my_run
```

## Architecture Notes (agents likely to miss)

### Entity Resolution staging
- Stage 1: embedding + vector storage. Stage 2: blocking + clustering (recall-oriented). Stage 3: LLM validation + graph rewiring (precision-oriented).
- `--store-backend memory` is ephemeral — you **must** use `--stage all` in one command when using it.
- Stages communicate through persisted artifacts, not shared in-memory state.

### Two merge paths
- **Online merge**: `POST /api/entity/merge` in backend, rewires the live Neo4j graph. Use for production corrections.
- **Offline merge**: entity-resolution stage3 rewrites JSON graph artifacts before import. Use for pipeline processing.

### RAG mode prerequisites
- `semantic_search`: needs Qdrant running + markdown collection created + indexed.
- `graph_search` / `naive_grag`: needs Neo4j running + graph imported.
- `hybrid`: needs **both** Qdrant markdown indexing and Neo4j graph data (combines semantic retrieval with deep graph_search).

### Shared infrastructure
- LLM access goes through `services.llms.get_llm(...)` provider abstraction.
- Neo4j driver is centralized in `apps/graph_api/neo4j.py` and reused by Graph API, RAG graph storage, and GraphSearch reasoning.
- Qdrant is used in two separate contexts: entity resolution vector backend and RAG markdown retrieval store.

## Coding Style
- Python 3.10+, 4-space indentation, `snake_case` functions/modules, `PascalCase` classes and Pydantic models.
- React: `PascalCase` component files, route views in `pages/`, API calls in `src/services/api.js`.
- Keep API routers thin; put reusable logic in `services/`.
- Use `requirements.txt` as the source of truth for Python dependencies; `pyproject.toml` only defines package metadata/build config.

## Thesis Work
- When working under `thesis/`, follow the thesis-specific rules in `thesis/AGENTS.md` and `thesis/THESIS_RULES.md`.
- Treat the thesis as a research artifact, not a production platform: prioritize clarity, reproducibility, explainability, and evaluation value.
- Avoid adding abstractions, dependencies, features, diagrams, or low-level implementation details unless they support the thesis objective or evaluation.
- For thesis figures and diagrams, follow `thesis/image/README.md`.

## Security
- Do not commit secrets. Copy `.env.example` files for setup. Keep keys in `.env`.
- Key env vars: `OPENAI_COMPATIBLE_API_KEY`, `OPENAI_COMPATIBLE_BASE_URL`, `OPENAI_COMPATIBLE_MODEL`, `NEO4J_URI`, `NEO4J_USER`, `NEO4J_PASSWORD`.
